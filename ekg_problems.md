## V тАФ VLM Output Improvements (for LLM Commentary)

_Grounded in `skill-commentator.md` ┬з "Why T-Box requirements differ for LLM vs SPARQL" and the `context_to_text()` output format._

The LLM commentator builds its prompt from `event_to_context()`, which reads these VLM-contributed fields:
`hasDescription`, `detectedJersey`, `hasEventType`, `hasTime/hasMinute/hasPeriod`.
When player name is missing (95% of events), `detectedJersey` is the **only fallback** тАФ
the system prompt says _"say 'the player' or use the jersey number if available"_.
And `hasDescription` is the **only visual context** the LLM gets for VLM-only events (no ESPN text).

| ID | Severity | What's missing | Why it matters for commentary | Where to fix |
|----|----------|---------------|-------------------------------|-------------|
| V01 | Critical | `hasDescription` is too thin тАФ VLM produces 1-sentence descriptions like "Player in red kit takes a shot" with no location, no technique detail, no scene context | The skill flags "NO TEXT" as a thin-event blocker; `context_to_text()` outputs `VLM: {description}` as the primary visual input to the LLM тАФ a weak description produces generic commentary | `action_recognizer.py` тАФ rewrite the VLM prompt to ask for: **who** (jersey#), **what** (action type + technique), **where** (pitch zone: penalty box / edge of area / midfield / own half), **how** (left foot / right foot / header / tackle type) |
| V02 | High | `detectedJersey` unreliable тАФ VLM misreads jersey numbers from 8 evenly-sampled frames, often returning wrong or null jersey | Skill system prompt: _"use the jersey number if available"_ тАФ with 95% unidentified players, `detectedJersey` is the last resort for naming anyone; wrong jersey = wrong name fallback | `action_recognizer.py` тАФ ask VLM to read the number from the frame where the jersey is most visible (highest resolution / closest to camera); optionally cross-check against `hasJerseyNumber` roster |
| V03 | High | No `pitchZone` field тАФ VLM doesn't report where on the pitch the action happened | Commentary gold: "he's struck from the edge of the area!" vs "a speculative effort from halfway!" тАФ this context cannot be inferred from any other field | `action_recognizer.py` тАФ add `pitch_zone` to JSON output (values: `penalty_box`, `edge_of_area`, `midfield`, `own_half`, `wing`); store as new datatype property `hasPitchZone` on Event |
| V04 | High | No `bodyPart` field тАФ for shots and goals the VLM doesn't report left foot / right foot / header as a structured value | Commentary: "a thunderous header!" vs "curled in with his left" тАФ currently this may appear in `hasDescription` text but unstructured so the LLM may or may not use it | `action_recognizer.py` тАФ add `body_part` to JSON output for Shot/Goal events; store as `hasBodyPart` on Event |
| V05 | Medium | No `foulType` field тАФ for Foul events VLM only says "Foul" with no sub-type (tackle / handball / push / shirt-pull) | Commentary tone depends heavily on foul type: "a cynical shirt pull" vs "a reckless sliding tackle"; also feeds into `TRIGGERED` severity decision | `action_recognizer.py` тАФ add `foul_type` to JSON output for Foul events; store as `hasFoulType` on Event |
| V06 | Medium | No `secondPlayerJersey` тАФ for tackles, duels, assists VLM doesn't report which other player was involved | Enables VLM-sourced `ASSISTED_BY` edges (currently only ESPN text produces assists); commentary: "played in by #7" | `action_recognizer.py` тАФ add `second_jersey` to JSON output for Foul/Assist scenarios; use it in `kg_builder.py` `ingest_matched_event()` to write `ASSISTED_BY` when present |
| V07 | Medium | `team_color` тЖТ team mapping unreliable тАФ kit color collision (P04) causes wrong team attribution; `teamSide` (home/away) is never explicitly stated | Skill's `context_to_text()` outputs `PLAYER: Name (Team)` тАФ with unknown player, at minimum we need the correct team; wrong team attribution makes commentary factually wrong | `action_recognizer.py` тАФ change prompt to output `team_side: "home" or "away"` based on kit colour + attacking direction (which goal they're moving toward); store as `hasTeamSide` on Event |
| V08 | Low | No `ballVisible` flag тАФ VLM doesn't report whether the ball appeared in any of the 8 sampled frames | The skill debug checklist implies thin-context events (no player, no text) should be flagged; a detection with no ball visible has low reliability and should be confidence-penalised before entering the buffer | `action_recognizer.py` тАФ add `ball_visible: true/false` to JSON output; use in `buffer.py` to apply a confidence penalty (`confidence *= 0.5`) when `ball_visible=false` |
